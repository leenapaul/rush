11 ## Introduction {#sec-parameters-list-intro}
12 
13 Parameters provide variables for a build, such as site name, repo information,
14 database connection strings, and really anything needed to complete a build.
15 
16 Any parameter may be overwritten.
17 
18 Parameters may be set but not used by every job.

40 ## build Parameters {#sec-build-parameters}
41 
42 Build parameters should be set in the params file within each Rush
43 job folder. However, it may be useful to also set some Build parameters at
44 the Environment level. For example, Rush defaults `docroot_name` to
45 'docroot', but all your builds may use 'www' as their docroot. You can then
46 set `docroot_name` at the Environment level to `www` so the default will now
47 'www'.

85 @param [job] => Test.params
86 <div>The name of the job. This parameter is just informational.
87 A period in the name indicates the job `params` is a sub-job of `Test`.
88 Derived: From the name of the job directory.
89 </div>
90 @param [key] => 08c17
91 <div>A unique id for the build. Used in test jobs. May be also be used for
92 other purposes such as to append to directory names to keep them unique or
93 in logging. Derived: Auto generated.
94 </div>
95 @param [domain] => params.dev
96 <div>The domain to use for the build if the job builds a site.
97 Derived: From a combination of the job name and the env tld parameter.
98 </div>
99 @param [machine_name] => params_dev
100 <div>A machine name for the build. The machine name has no whitespace and no
101 periods in it. The machine name is typically the basis for other parameters
102 such as the db name, if none was provided.
103 Derived: From the domain name with spaces and periods removed.
104 </div>
105 @param [site_uri] => http://params.dev
106 <div>Either a uniform resource locator (URL), or a uniform resource name (URN),
107 or both. Usually just the url to the build site.
108 Derived: From `http://` plus the domain name. If the host post is not set to
109 the default of `80`, then the host port will be added. This is mostly used for
110 MAMP users with a non-standard port setup. So for example, if the port is set
111 to `8888` and the domain is `params.dev` then the URI will become
112 `http://params.dev:8888`
113 </div>
114 @param [root_directory_name] => Sites
115 <div>
116 The name of webroot directory to use for the build.
117 Example settings include: 'www', 'websites', and 'Sites'.
118 Derived: 'Sites' as a default; which is kind of OSX-centric.
119 </div>
120 @param [root_directory_path] => /Users/username/Sites/
121 <div>
122 Most often set to the path to the webroot directory on the server. Usually it's
123 something like `~/webroot` or `~/Sites`. NOTE: it does not have to be set to
124 the webroot directory. A particular Drush Rush job may just download a bunch of
125 code libraries to a directory somewhere, so the path is often set globally but
126 may be overwritten at the job level. Derived: From the Rush Server Home plus
127 the root directory name plus a forward slash.
128 @code
129 $params['build']['root_directory_path'] = $params['global_locations']['RUSH_SERVER_HOME'] . $params['build']['root_directory_name'] . '/';
130 @endcode
131 </div>
132 @param [logs_path] => /Users/username/Sites/logs/
133 <div>
134 If Rush detects you have a directory named `logs` within your root_directory_path
135 then Rush will add an `ErrorLog` entry in your vhost file pointing to the directory.
136 You may also set `logs_path` to another directory of your choice.
137 You can set it at the global level in your environment folder or at the job level.
138 </div>
139 @param [directory_path] => /Users/username/Sites/params_dev/
140 <div>
141 The build_directory_path is the sum of the build_root_directory_path and the
142 build_machine_name. It's automatically derived. However, setting this value in
143 a job params file will override the path. In practice, this would be the only
144 value needed to be overwritten, rather than the build_root_directory_path and
145 the build_machine_name. Derived: From the root directory path and the machine name
146 plus a forward slash.
147 @code
148 $params['build']['directory_path'] = $params['build']['root_directory_path'] . $params['build']['machine_name'] . '/';
149 @endcode
150 </div>
151 @param [docroot_name] => docroot
152 <div>
153 A Drupal project may contain directories above its html / docroot. Files meant
154 to be accessed sub-directory of the overall project. Drush Rush must know where the core
155 Drupal code is. Example values include: 'htdocs', 'www', and 'docroot'.
156 Drush Rush defaults to 'docroot'.
157 </div>
158 @param [docroot_path] => /Users/username/Sites/params_dev/docroot/
159 <div>
160 Used by Drush and Apache Vhosts. Path to Drupal core code base. Derived: From
161 directory path and docroot name plus a forward slash.
162 @code
163 $params['build']['docroot_path'] = $params['build']['directory_path'] . $params['build']['docroot_name'] . '/';
164 @endcode
165 </div>
166 @param [code_directory_path] => /Users/username/Sites/params_dev/docroot/
167 <div>
168 Typically an alias for the docroot_path. However, code_directory_path can be
169 changed if the codebase is located somewhere else.
170 </div>
171 @param [site_folder_name] => default
172 <div>
173 The name of the Drupal site folder, inside the sites directory. Defaults to: `default`.
174 </div>
175 @param [site_specification] => /Users/username/Sites/params_dev/docroot/?uri=http://params.dev#default
176 <div>
177 A Drush site specification. Derived: From docroot path, plus uri, plus # sign
178 and the site folder.
179 </div>
180 @param [site_alias_record] => Array
181 <div>
182 A Drush site alias record. Derived: Provided by Drush.
183 @code
184 Array = (
185  [uri] => http://params.dev

187  [path-aliases] => Array
188  (

194 </div>

213 ## db Parameters {#sec-db-parameters}
214 
215 Database parameters may be set in the params file within the Rush
216 application folder `~/Rush/environment/`. The DB parameters may also be set
217 at the Job and SubJob directory level.
218 
219 @param [query] => UPDATE users SET mail = 'admin@example2.com' WHERE users.uid = 1
220 <div>
221 A string containing a query to run on the database with the `sql_query` operation.
222 </div>
223 @param [type] => mysql
224 <div>
225 The type of database server.
226 Options include: `mysql` , `sqlite` , and `postgres` .
227 Currently only mysql has been tested.
228 The database type is used to create a db connection url.
229 Defaults to: `mysql`.
230 Meant to be set in the `environment` directory and overwritten at the job level when required.
231 </div>
232 @param [server] => localhost
233 <div>
234 The db host server identifies the server address to connect to.
235 It identifies either a hostname, IP address, or UNIX domain socket.
236 Examples include: `localhost` and `127.0.0.1` .
237 The default is `localhost`.
238 Meant to be set in the `environment` directory and overwritten at the job level when required.
239 </div>
240 @param [port] => 3306
241 <div>
242 The database port is the port on which the database server is listening for
243 connections. The default mysql port is `3306`. MAMP users may use a different
244 port, such as `8889` .
245 Meant to be set in the `environment` directory and overwritten at the job level when required.
246 </div>
247 @param [user] => root
248 <div>
249 The database user name and password are used to create a db connection url.
250 All db values may be changed at the Job level for jobs which need to use
251 different databases and database connection information.
252 Both default to `root`. Please note: Putting user names and passwords in text
253 files has security risks. Rush should only be used on local secure systems
254 behind a safe firewall.
255 Meant to be set in the `environment` directory and overwritten at the job level when required.
256 </div>
257 @param [password] => root
258 <div>
259 The database user name and password are used to create a db connection url.
260 All db values may be changed at the Job level for jobs which need to use
261 different databases and database connection information.
262 Both default to `root`. Please note: Putting user names and passwords in text
263 files has security risks. Rush should only be used on local secure systems
264 behind a safe firewall.
265 Meant to be set in the `environment` directory and overwritten at the job level when required.
266 </div>
267 @param [name] => params_dev
268 <div>
269 The name of the database to use for the job.
270 The db name will usually be set in the project params file.
271 If it's not set, then it will default to the job machine name, which in turn
272 may be derived from the job domain.
273 </div>
274 @param [url] => mysql://root:root@localhost:3306/params_dev
275 <div>
276 A database connection URL used to connect to the build database.
277 The db url is built for you based off the following other db params:
278  - type
279  - user
280  - password
281  - port
282  - server
283  - name
284  - file_path (for sql lite)
285 <div>
286 You can set the db url in your job params to bypass the derived version and to
287 skip the need to set other db params. This is helpful if your build should go
288 to a different db connection than what's default on your system.
289 @code
290 MySQL
291 url = mysql://root:pass@localhost:port/dbname
292 
293 SQLite
294 url = sqlite://sites/example.com/files/.ht.sqlite
295 @endcode
296 </div>
297 </div>
298 @param [file_path] => /Users/buscay/Sites/params_dev/sql/params_dev.sql
299 <div>
300 The `directory_name`, `directory_path`, `file_ext`, `file_name`, `file_path`,
301 db parameters are used to backup and import an sql file into and out of the build.
302 Each parameter builds upon the previous parameter settings to complete the
303 `file_path` parameter. If you need to point to point to a db file other than
304 what is automatically derived by Rush, then you can change one of these
305 parameters to get what you need.
306 @code
307 [directory_name] => sql
308 [directory_path] => /Users/buscay/Sites/params_dev/sql/
309 [file_ext] => sql
310 [file_name] => params_dev.sql
311 @endcode
312 <div>
313 Here are some examples:
314 - The default db `directory_name` for builds is `sql` , but all your builds
315  store the db file in a build directory called, `db_snaps`. Simply change the
316  `directory_name` name parameter at the environment level to `db_snaps` to
317  have all jobs export db snapshots and import dbs from a directory called,
318  `db_snaps`.
319 - The default db `file_ext` is `sql` , your you have one job which requires
320  the extension to be `mysql` . Change the `file_ext` parameter in your job
321  params file to `mysql` .
322 - For a particular job build, you need to import the staging database rather
323  than the default database. The staging database is called, 'staging.mysql'.
324  Change the `file_name` in the job params file to `staging.mysql` .
325 - All default parameters will point your job to build a site using a db
326  snapshot located here: `/Sites/mysite_dev/sql/db.sql` but you want the job
327  to build using a db located here: `/Desktop/migrate.mysql`. Just change the
328  `file_path` parameter to `/Desktop/migrate.mysql` and this will supersede
329  all other db location and name parameters.
330 </div>
331 </div>
