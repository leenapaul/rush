13 ## Purpose {#sec-purpose}
14 
15 The project's purpose is to allow Drupal Developers to:
16 
17 - Reduce typical Drupal development workflows down to one step (one Drush command).
18 - Create re-usable and share-able Drupal development operations and workflows.
19 
20 ## Project Status {#sec-status}
21 
22 Currently in Beta. Ready for initial public release.

24 ## Background {#sec-background}
25 
26 This project originates out of experience using Jenkins to automate building and testing Drupal projects.
27 
28 > **Note:** Jenkins is an extendable open source continuous integration server. Visit: http://jenkins-ci.org/ for more information.
29 
30 Beyond its excellent tool set to execute and monitor the results of repeated build jobs, Jenkins is a good tool to automate repetitive workflows for developers.
31 
32 For example, if a developer needs to work on a website locally, he or she typically must:
33 
34 1. Get the code via a Versions Control System like git
35 2. Switch to a branch or tag to work on
36 3. Change or add some parameters in settings files to match the development environment
37 4. Get the db export
38 5. Create a testing db
39 6. Import the db snapshot
40 7. Create a local host file entry
41 8. Restart Apache
42 
43 That's eight steps before starting to code.
44 
45 Jenkins can automate many of these common developer workflows, using job parameters to let developers simply enter a project name and a branch and then run a Jenkins job to do their repetitive build work for them.
46 
47 Like Jenkins, Drush Rush is a tool to run parameterized builds with Drush.
48 In most cases the build jobs are a series of core drush commands chained together to complete a full build process with a single command.
49 
50 ## A Few Points about Drush Rush Jobs {#sec-about-jobs}
51 
52 1. Drush Rush jobs are **written once then used repeatedly**. *There is probably little value to writing a Drush Rush job for a one time build process.*
53 2. Drush Rush jobs are **extendable**. Complex jobs may be written by extending and or combining similar jobs.
54 3. Drush Rush jobs are **shareable** so a job written by one developer may be shared with many.
55 4. Drush Rush jobs are **light weight**. Drush Rush may be used to execute steps as part of a continuous integration or automated testing process, however, it is not a CI or Testing Suite.

61 ## Example Usage {#sec-example}
62 
63 Run a Drush Rush command like:
64 
65  drush rush myjob.init
66 
67 To parse the job folder's rush files for build parameters and build operations and then execute the following operations:
68 
69 1. Make the build directory
70 2. Put the directory in the web root
71 3. Add other directories and files to the build (setup the build skeleton)
72 4. Create a docroot folder for the Drupal code base
73 5. Build the code base with Drush Make using specified makefile(s) and prepare the site for install
74 6. Create the specified database
75 7. Create the specified dns and virtual host entries
76 8. Run any pre-build commands
77 9. Run any pre-build scripts
78 10. Create build aliases.drushrc.php files
79 11. Install the site
80 12. Change variable values as specified
81 13. Run any post-build commands
82 14. Run any post-build scripts
83 15. Put the build under version control
84 16. Create a remote repo (such as on GITHUB), commit the code, and push it
